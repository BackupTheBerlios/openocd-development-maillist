On Monday 03 March 2008 21:59:04 Øyvind Harboe wrote:
> Trying to make some progress on this one.
>
> We all want the code to be correct and we all want
> to find those bugs.
>
>
> 1. Add a new fn umalloc(). Alloc of uninitialized memory.
>
> 2. Add command line option which controls uninitialized memory:
>   a. zero (default) clears memory. Broken code will behave identically
>      on all machines. There is a good chance that zeroing
>     will do the right thing, which is bad in terms of finding
>     the problem, but good for users. There is a relatively good
>    chance of a SEGFAULT upon dereferencing NULL pointers.
>   b. fill=0xnn. Fill value intended to provoke problems, default 0x55.
>   c. uninitialized. For use w/valgrind and such.
>
> 3. This option can be used in the testsuite and by developers.

Ok. There's already something along those lines in helper/replacements.h IIRC.

It just has to be made clear that umalloc is _only_ correct when it replaces a 
malloc. It's perfectly valid for code to use calloc if it wants zero 
initialized memory.

Dominic

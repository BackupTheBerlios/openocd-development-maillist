- statemove is now internal to the driver.
- all drivers are *required* to implement pathmove, even if that means that it
would only report an error. If a driver can not support general path moves,
it may be able to support some pathmoves using internal statemoves,
but the driver needs to sort this out.
- xsvf now uses pathmove since it has *very* particular requirements
on the path used and there is no room for the driver to try to squirm out
of a particular pathmove sequence. xsvf needs work and this is the
beginning of a fix.
- application code should not use jtag_add_pathmove() unless it
has to, but it should if it has to. code that requires, but does not use,
jtag_add_pathmove() may happen to work on some interface, but not others.

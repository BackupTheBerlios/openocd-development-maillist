On Fri, Mar 28, 2008 at 2:33 PM, Dominic Rath <Dominic.Rath at gmx.de> wrote:
> We've talked about interfaces that can't implement arbitrary path moves before, so
> please don't remove statemove.

It is not removed. It's still in use by the driver, it is just not
exposed in the
same manner.

>  TI for example has a JTAG TAP master that suffers from this limitation.

As jtag_add_pathmove() is defined it is *only* to be used when some guarantee
is required as to the precise path to be taken is needed. Here the TI JTAG TAP
would be a non-starter anyway.

Also the TI device *can* implement pathmove, it just needs to return an error
if it isn't able to honor the path requested.

>  I'll update the jtag.h comments to reflect the path requirements as
> described by SVF (TI implements those, afaik).

Statemove still exists at the driver level. We don't need to expose this
in the API.

Certainly unless testers step up, devices such as the TI JTAG master will
never work with OpenOCD.

Are there any testers out there today of such devices?

Ref. earlier discussions support for devices is more about testing & testers
than it is about programming.

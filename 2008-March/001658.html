> We've talked about interfaces that can't implement arbitrary path
> moves before, so please don't remove statemove.

Forcing the seven-step state moves seems to violate the (X)SVF standard. 
   So, we have a real contradiction here.

While the "extra steps do not hurt if placed carefully" may be 
relatively safe with ARMs, I would not count on it with all sorts of 
programmable chips. SVF can be used for very different applications.

The only safe solution is to use pathmoves only in SVF and to refuse to 
program (or at least ask for confirmation) if the user tries to use 
(X)SVF files with a seven-step programmer.

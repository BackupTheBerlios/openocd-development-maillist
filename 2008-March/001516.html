Hello Øyvind,

Øyvind Harboe wrote:
> I'd like the telnet and GDB server to be available before the JTAG
> device chain has been enumerated and validated.
> Any objections to that goal?

No problem with the telnet, but I am not sure about GDB.
OpenOCD supports multiple targets and it distinguishes between targets by TCP port number GDB connects to.
That is, unles a target is defined, OpenOCD does not listen on any port for incomming GDB connections in the present state.

I am not sure how to make GDB  available even without a target configured
so that it behaves logically. I suggest to take cautious steps in this, lets go just with the telnet first.


> The reason is that e.g. GDB init scripts can use monitor commands
> to initialize the target.  This matches expectations of a lot of users
> that have used other hardware debug solutions.

Although it does not seem so crucial me, it is probably a good to have this.


> Other advantages is that the target might only be powered up
> after OpenOCD is up and running.

Well, probably not a big deal. The previous goal seems much more interesting to me.


> I'm hopeful that the ramifications won't be too severe.  I don't
> expect any regressions/colateral damage.

Well, I am not absolutely sure about this, see above, at least it changes the logic
of behavior with multiple targets (?).


> Mostly it should
> involve making OpenOCD more robust with flaky JTAG
> connections.

That would be good ;-)

On Sunday 02 March 2008 11:11:22 Michael Bruck wrote:
> This project is most definitely *not* ANSI C. Not even close. None of
> the variadic DEBUG() etc. macros would work under ANSI C. The function
> pointer struct initializations aren't ANSI C. And lots of other things
> aren't. The ANSI C mantra is just a distraction to impose silly and
> arbitrary coding puritanism on others.

See Øyvind's "ANSI C vs C99" post. The OpenOCD isn't ANSI C, but rather ANSI C 
+ ISO C + gcc enhancements + a bit of C99.

> When I send patches to fix actual bugs I try to keep the formatting of
> the surrounding document. Is it asking too much that people refrain from
> 'beautifying' the code I have to work with regularly as if my coding
> style choices are somehow random and silly?

I guess the answer is yes. Like most other open source projects the OpenOCD 
tries to follow some coding standard, and its certainly less restrictive than 
many other projects. The goal of such codings standards is to improve code 
readability, and to ease code transfer from one part of the project to 
another.

> W.r.t. the ARM11 code, it would be so nice if someone had a substantive
> contribution or even just bugfixes or maybe advice on the interaction
> with the back-end to offer, instead of reformatting the files.

That's difficult, because ARM11 hardware isn't easy to get. This will 
certainly change once more ARM11 designs are available like the ARM9s are 
available today.

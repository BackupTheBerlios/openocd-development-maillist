> jtag_add_statemove() is now a no-op if state == current state,
> except state == TAP_TLR which always resets the state machine.

The TAP_TLR thing is a good thing, but there may be a problem with the 
no-op.

I may be wrong, but at least the SVF (not X-) spec says that state moves 
from a state to the same state have to be carried out through the 
standard path. For example, DRPAUSE-DRPAUSE has to take 
DRPAUSE-DREXIT2-DRUPDATE-DRSELECT-DRCAPTURE-DREXIT1-DRPAUSE. I do not 
have the XSVF spec in hand, but IIRC it should behave similarly.

(And if you already have taken this into account by using explicit state 
moves, then sorry about the noise.)

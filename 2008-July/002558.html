> I think it is important that all commands return very soon (from the human
> point of view). If a command takes long (such as a large memory read/write),
> it should not block, as that makes HMI design more difficult. It is much
> easier to poll than to create a number of threads and to wait for a possibly
> very slow command to succeed or poll them with another connection. The
> timeout for a human being is in the order of some hundreds of milliseconds
> (and for a seasoned nerd much less).

I'm loathe to place this responsibility on the scripting language. It
complicates
the model.

I'd rather make it the responsibility of the client to implement support for
this. This can be done via multithreading or using a post + check for
reply concept.

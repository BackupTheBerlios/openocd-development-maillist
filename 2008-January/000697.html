>> I would not call probe for every flash bank every time we look for a

> In my posting I wrote about explicit vs. implicit probe. Fixing
> the crash took priority, caching comes later. Also it is important
> to be able to exercise a probe in all the places where a
> probe could happen w/caching, so even with caching there
> would have to be a way, for testing purposes, to disable
> that caching.
>
> Explicit probe would be a "flash probe" command. Here the probe *should*
> happen every time, alternatively the "flash probe" command could
> simply be removed w/autoprobe.
>
> Without a consensus that OpenOCD *should* support
> autoprobe, this whole discussion is moot though.
>
>   
I definitley support  autoprobing but I feel that probing everyting 
everytime is a bit ugly.
>> There are situations where clock and status information must be checked
>> for every, or at least many,
>> flash operations ( stellaris.c ) but this can be handled in the  device
>> specific flash driver code.
>>     
>
> Please explain a bit more.
>
>
>   
The stellaris flash code needs to set timing parameters, those are 
influenced by the current chip pll settings.
So for all operations that relies on clock speed the clock information 
is probed, this is a part of the information
gathered in a full probe. My point was that this process can be local to 
the driver.
Of course the probe command reprobes everything including timing info.

On Jan 17, 2008 11:43 PM, Dominic Rath <Dominic.Rath at gmx.de> wrote:
> - There could be all kinds of hidden problems when allowing all commands to
> execute during configuration (Øyvind mentioned that change in some post). The
> daemon startup pretty much relies on CONFIG commands executing before EXEC
> commands.

Could this be mitigated by caching commands in three lists (CONFIG,
EXEC, ANY) before actually handling them? Basically, you would read in
all commands from scripts into the lists. Once you have read in all
commands, then walk the lists and call the handlers. Of course, you
would have to call the handlers for the "script" commands immediately
instead of caching them.

Doing this would remove the element of surprise if the user sets up a
script or config file with things EXEC commands before CONFIG
commands.

This may be naive on my part since I am still trying to grok
command/config processing. I thought I saw something in the code about
command_t being a linked list.

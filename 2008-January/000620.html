Hi,

I'm trying to get openocd working for use with project vga, which has  
3 xilinx devices in the chain. Unfortunately I'm getting quite a few  
errors when trying to run some basic xsvf files generated by iMPACT,  
so I started debugging a bit.

One error I seem to have solved, but 1) I don't know how to generate  
a proper diff, so sorry about that and 2) I'm not even sure if this  
is a proper 'fix'. The xsvf in question can be found here:
http://projectvga.org/other/test3.xsvf

Which, with revision 253 gives;
Error:   ft2232.c:477 ft2232_add_pathmove(): BUG: rti -> tlr isn't a  
valid TAP transition

According to:
http://www.xilinx.com/support/documentation/application_notes/ 
xapp503.pdf (page 21, bottom)

"If the state is 0x00 (Test-Logic-Reset), then force the TAP to the  
Test-Logic-Reset state via the
guaranteed TAP reset sequence: hold TMS High and apply a minimum of  
five TCK cycles. For
non-zero state values, if the TAP is already in the specified state,  
then do nothing. Otherwise,
transition the TAP to the next specified state."

Which made me change line 473 of ft2232.c from:
else if (tap_transitions[cur_state].high == cmd->path[state_count])
	buf_set_u32(&tms_byte, bit_count++, 1, 0x1);

to:
else if (tap_transitions[cur_state].high == cmd->path[state_count]  
|| !cmd->path[state_count])
	buf_set_u32(&tms_byte, bit_count++, 1, 0x1);

It might not be the prettiest way to get the desired effect, but it  
seems to work:
 > xsvf plain test3.xsvf
XSVF file programmed successfully

Since the requests some device IDs and compares them with what they  
should be, I assume it's executed correctly now -- I don't know how  
to confirm this for sure though.

Could anybody give me some pointers as to how to confirm this, or if  
all this is correct? I've got a couple more error messages when I try  
to do other stuff (like request the device id of the first device -  
it captures the device id of the last, then aborts because of that)  
and want to know if I'm on the right track before continuing.

Thanks,

Mike
www.projectvga.org

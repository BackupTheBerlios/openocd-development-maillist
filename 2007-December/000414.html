OpenOCD+GDB will fail to evaluate stack traces using the MI command
interface("info stack" works)
when OpenOCD returns error message w/data abort to GDB under certain
circumstances.

Eclipse CDT uses the MI command interface(for good reasons).

Possible solutions:

1. fix GDB to return as much of the stack that it was able to evaluate(which
basically means the entire stack trace). It should print a warning, but not
fail, in this case.
2. modify OpenOCD to lie and return some value & no error code so as to
let GDB succeed.
3. add hacks in CDT to be able to cope w/the situation
4. The developer can fix his app(e.g. eCos app).

Comments:

1. the correct solution and also the most difficult to get done. It
would help *all*
hardware debuggers out there.
2. this is a hackish, but expedient solution. I think this is what
other hardware debuggers
do.
3. This is a bottomless pit of missery. CDT will never get this quite right.
4. This is make-work for the application developer IMHO, also I think
he'd throw out
the hardware debugger in question and use one that doesn't exhibit this problem
given a choice.

More info:

https://bugs.eclipse.org/bugs/show_bug.cgi?id=86676


Does anyone else see this problem? Anyone else with an opinion as to
what would be the
best solution?

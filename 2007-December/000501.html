Øyvind Harboe wrote:
> The flash are slow to the point of useless without working area... I'm
> not quite sure if
> it makes any sense for OpenOCD to support e.g. CFI drivers without working area
> except for OpenOCD developers testing purposes.... Perhaps it would be better if
> OpenOCD refused to set up e.g. the CFI flash driver without working area.
>   
Strong NAK. On some targets[1], setting up a working memory area can be 
*really* painful, so to develop a bootloader (or to reflash a broken 
one), flashing without working_area is the only possibility, even if it 
is painfully slow.

What is more interesting IMHO is the comment that the insight cable 
(what cable is that?) can do the same job much faster - maybe openocd 
can be improved up to that speed, too?

cu
Michael

[1] I am thinking about modern ARM9/ARM11 SoCs with DDR2 memory 
controllers and no integrated SRAM, where setting up the DDR2 memory 
controller requires complex calibration code, and reading of SPD EEPROM 
values - I have worked on a Marvell 88F5181 for a short time (using a 
BDI2000, not openocd), and quickly decided that flashing without a 
working area was the only practical way until the bootloader can 
initialize RAM.

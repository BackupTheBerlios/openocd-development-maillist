On Thursday 09 August 2007 14:35:16 Pavel Chromy wrote:
> That is exactly the problem I was facing, finally I have chosen a simple
> way - just to lookup flash bank according given address like if all were in
> the same address space. This is not a perfect solution in theory, but works
> well in a typical situation.
>
> The approach I took:
> The 32-bit virtual address space is huge (when speaking about MCUs) and
> even if some flash bank is in fact not part of the targets address space,
> it may be mapped into some unused range in this address space (for
> programming purposes). In my opinion, this is not a significant limitation
> for practical usage and we can live with that before there is a better
> solution.
>
I know of at least one user who's debugging multiple LPC2000 devices on a 
single board that are part of the same JTAG chain. All of those targets will 
have their flash mapped to the same address. This might be a corner case, but 
it's definitely possible.

> Maybe a silly question:
> By the way, what about Harvard architectures? I am not aware how
> OpenOCD/GDB handles that as I am not familiar with ARM9 yet - just started
> with ARM7 which is a Neumann.

While ARM9 is a hardvard architecture hardware wise (separate busses for 
instruction and data memory) both memories are usually accessible from either 
bus, though ocassionally with performance penalties. From a programmers 
perspective the ARM9s usually appear as a von-Neumann machine.
Anyway, if flash programming is possible at all from within the target there 
needs to be a way to access the flash via the data bus, therefor we're only 
concerned with one address space, even if they're actually separate.

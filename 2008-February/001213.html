On Thu, Feb 28, 2008 at 01:10:46PM +0100, Øyvind Harboe wrote:
> On Thu, Feb 28, 2008 at 1:07 PM, Uwe Hermann <uwe at hermann-uwe.de> wrote:
> > I added a file doc/TESTING, which is a plain-text version of
> >  testing/index.html (HTML documentation is a hassle IMO), and also
> >  fixed a few typos etc.
> 
> There are advantages to html, so I'm holding off commit on this one.
> 
> - tables
> - links to explanation of tests
> - images

Yeah, sure, but that kind of stuff is more appropriate for the wiki
anyway. SVN stuff shouldn't be HTML, IMHO.


> >  I moved the document to doc/ where it belongs, the committer should do
> >
> >   svn add doc/TESTING
> >   svn rm testing
> >   svn ci
> >
> >
> >  Regarding the text "Commit, then test" I disagree, that's a bad idea.
> >  It should rather be "Test _before_ committing".
> 
> Explain.
> 
> The crucial point here is that we must know *what* we tested. Something can go
> wrong in the commit process.

Hm, well, that's a mistake then. You'd have to re-run the tests if you
mess up the commit. But that should rather be the exception.

The proper way would usually be:

 1. Create patch
 2. Apply patch (do _not_ commit)
 3. Run tests
 4. Post patch and your test results to list, wait for comments
 5. If nobody complains or comments after a while, commit

(or something along those lines)


The procedure of "commit first, test later" is what I think is a bad idea.
You want to always keep svn in a working, building state.


> E.g. testing can happen over many days at different developers.

Sure, no problem. Everybody can apply the patch and then test. No need
to commit the patch until it's been well-tested.

Hello Øyvind and the list,

Øyvind Harboe napsal(a):
> - have a library of packaged target and device scripts. They can either be
>   used as-is or as templates.
> - split target and debugger config scripts. A target config script
>   contains only information about the target and the debugger device
>   config script only about that device debugger.
> - Add command line option to choose device and target
>   config script. Something along the lines:
>   openocd --device ft2232 --target at91eb40a

I am not sure whether it is really necessary to force users of OpenOCD into
one or another scheme of configuration files/scripts.
It is certainly a good idea to have library of examples or templates,
but for this, I think it is sufficient if OpenOCD accepted several
configs/scripts on its command line, like this:

openocd --file ft2232.cfg --file at91eb40a.cfg

This way the users may use ever finer separation of configuration data,
e.g. device, board, processor and external flash.
I am simply not convinced about benefits of implied differentiation
of purpose of particular configuration file.
A generic way to achieve the same effect seems better to me
- just let the user decide how to structure his/her cofigs and scripts.

Good thing isability to include a config file from another one
(BTW, does this work already? I remember there was a discussion about using "script" command from a config file.)

Oh, and when speaking about this - I am for unifying config files and scripts - steps to this were already taken
by allowing any commands to be used in config files, everything would be just "scripts".

And finally it would be nice to have a possibility to invoke openocd command directly from command line,
like this:

openocd --file ft2232.cfg --command "jtag_speed 3" --file at91eb40a.cfg
in which case OpenOCD would process ft2232.cfg, then set jtag speed and finally process at91eb40a.cfg

Regards,

> Hi Øyvind,

> I see your point. I tested two options. A "reset run_and_halt" is asynchronous. A "reset init"
> seems to execute the reset script completely before the next command is accepted.
>
> My feeling is that the asynchronous behavior is dangerous, because it is unexpected.
> I expected the target to be fully initialized before executing the next command, e.g.
> loading the program into SDRAM.

I agree. My thinking is that reset should always be synchronous. Why
would anyone want
it to be asynchronous?

There needs to be a timeout w/error message of course if it is asynchronous.

> Can the behaviour be changed to synchronous, or can there be a command
> to wait until a script has finished?

There is such a command already: wait_halt.

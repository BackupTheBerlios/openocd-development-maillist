Even if you don't care about Eclipse(perfectly understandable),
you should care about bugs that Eclipse unearths.

View it as a good tool to exercise some lesser throdden
OpenOCD code paths.

One of the major outstanding problems with Eclipse and OpenOCD
is that memory read packets are handled *very* inefficiently by
OpenOCD and this breaks the 2 second default timeout. There
are a handful of other OpenOCD bugs that make this much
worse.

These problem exists with *all* GUI debuggers, but it is especially
bad with Eclipse since Eclipse, in it's wisdom and helpfullness,
will evaluate variables upon mouseover events! I tried to use
Insight w/OpenOCD and it imploded after a couple of
minutes on the same test code.

If you combine this with -O3, where pointers frequently will(this
is not a bug in GCC) point to outer space(i.e. memory regions
that will yield a data abort exception) combined with the
lack of checking the error value returned from target_read_buffer(),
then OpenOCD will simply appear to have crashed. If it walks like
a duck, quacks like a duck, then any user would call it a duck.

On Thu, 21 Feb 2008, Øyvind Harboe wrote:

> On Thu, Feb 21, 2008 at 6:53 AM, Nicolas Pitre <nico at cam.org> wrote:
> > If arm7_9_execute_sys_speed() times out, that probably means single-step
> >  never returned to debug mode.  Unconditionally calling
> >  arm7_9_debug_entry() in that case produces really interesting but
> >  completely foobar results as well as making OpenOCD really confused.
> >  The attached patch fixes this.
> 
> Makes sense to me.
> 
> There are a lot of places in OpenOCD where the return error code is ignored
> where it probably shouldn't be. E.g. XScale is pretty much guaranteed
> to get stuck in loops w/failed communication when e.g. the JTAG
> communication is flaky. Making OpenOCD respect return error
> codes *everywhere* won't happen overnight, but a few choice
> fixes as above will do wonders for stability.

Right.  This one was the result of real life experience.

I also have a similar issue with arm926ejs_read_cp15() and 
arm926ejs_write_cp15() which contain a do { } while() loop that never 
exits in some... hrm... circumstances.  Killing openocd is the only way 
out in those cases.

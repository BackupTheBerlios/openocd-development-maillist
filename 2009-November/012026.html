>> I've found some contributions to CMake support on the mailings list.
>> But I couldn't figure out if it will be supported and if 'Yes' then
>> when?
>
> The CMake support did not make into mainstream because we didn't
> want to maintain two sets of makefiles and maintainers here are mostly
> versant in autotools... The CMake stuff was good, but not quite done
> at the time.
>
> Now that the release system is pretty much in place, I believe that the
> autotools stuff in OpenOCD will be quite stable & tested.
>
> I don't know what would motivate us to try to add CMake support at
> this point.
>
> Even if we took it for granted that CMake was superior to autotools, that
> wouldn't be a reason to switch.

I find CMake in combination with CPack very useful to create
distributable packages. You can pull the latest git version of OpenOCD
and make a *.deb package from it and then deploy it on other stations
without needing to compile it on those machines. That would make the
life of hardware manufactures, that are going to package particular
version of OpenOCD to support their hardware, much easier.

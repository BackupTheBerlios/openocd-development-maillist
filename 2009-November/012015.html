Hello guys and thank you all for your answers! :-)

On Fri, Nov 6, 2009 at 8:52 AM, Øyvind Harboe <oyvind.harboe at zylin.com> wrote:
> We discussed the release numbers early this year.
>
> We wanted a numbering scheme that most users
> of OpenOCD would understand immediately without
> any explanation.
>
> 0.3.0 => the first 0 is meaningful. It conveys that we feel that
> it still needs a lot of work. Not only in terms of code but also
> process and community.
>
> 3 => the real release number
>
> trailing 0 => we have yet to cut a bugfix release as we intend
> to have short cycles.
>
> Lets keep things as-is unless there is wide agreement
> to switch to something *significantly* better.
>
> As cycles can be *long* in embedded development, there is
> nothing wrong with using a tool that is several years old, so
> I'm glad the date/year isn't in there.

Now I understand that "it is too early" to use the third digit
(micro), and the meaning of the first digit (major) if obvious.
However I think that using dates or -rc suffix or any other
modification of this numbering can bring confusion, that's all, no
offence :-)
So if the bugfixes are done on-the-fly with the code repository, and
the new functionalities arise meanwhile (and will constantly in
future), maybe it would be more convenient to produce releases more
frequently without the micro release part..? That's just my guess -
OpenOCD develops with a speed of thunder and I don't really believe
there will be any separate time for bugfixing and separate for adding
new features :-)

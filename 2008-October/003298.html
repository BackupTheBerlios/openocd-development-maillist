Timothy Clacy wrote:
> I'm not sure the details of setting-up SDRAM/DDR2 are too much of a
> problem. I would imagine many OpenOCD users are embedded system
> developers and have to write the SDRAM/DDR setup code anyway for their
> firmware. Also, I'm sure you would quickly build-up code sequences for
> popular platforms; some of your target scripts already do SDRAM setup
> using a short sequence of memory writes.
>   
I do write SDRAM setup code, and believe me, I would rather skip 
re-doing that in the form of a openocd init script if I can. On the 
IXP42x, getting the SDRAM up is quite easy, but on some DDR1/DDR2 
controllers I have seen that require DQS calibration depending on the 
board layout, it is a lot easier to skip that step and let the 
bootloader do the work.

However, to get the bootloader up and running, you need to write to 
flash - and writing with target code is a lot faster than without, so 
when the target code can be run from CPU-internal SRAM/cache, that means 
I can skip DDR setup and save a lot of work.
If necessary, it will need to be done, but I see no benefit in ditching 
the current, compact target code in favour of something bigger and 
slower that requires me to setup DDR memory before use.

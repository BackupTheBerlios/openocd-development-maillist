> michael> That depends. Getting DDR2 SDRAM controllers 
> initialized can be
> a quite tedious task,
> michael>  so you may be stuck with what internal SRAM you 
> have available.
> michael> The IXP42x has 2k mini-IC available for code, other CPUs may
> have even smaller TCMs

The PXA25x (and PXA26x?) processors also have a 2K mini-I-cache but,
because of the Harvard architecture, the code has to be very carefully
crafted; there is nowhere to load data from :-O

PXA27x has 256KB of internal SRAM so you could download a 'real' program
to the mini-I-cache and/or the SRAM.

> Often those big types of CPUs have CACHE which can be fixed at an 
> address and put in lockdown mode.

I don't know about other architectures but on Xscale, you can't lock
down cache until the MMU is enabled... which requires a translation
table... which usually requires big SDRAM for the translation table.


I'm not sure the details of setting-up SDRAM/DDR2 are too much of a
problem. I would imagine many OpenOCD users are embedded system
developers and have to write the SDRAM/DDR setup code anyway for their
firmware. Also, I'm sure you would quickly build-up code sequences for
popular platforms; some of your target scripts already do SDRAM setup
using a short sequence of memory writes.

I think that makes sense.

> I think that
>  could still be quite ok, as the input format is not difficult. Also,
>  that way the existing parser in OpenOCD could be used without
>  modification.

This would be a bit of frontend code to the parser, so the parser would
not be modified.

>  So, the changes need to be done to the way OpenOCD outputs things.
>  Most of the time we just need to have the success/failure status,
>  but there are commands which actually give some numerical output
>  (such as poll, reg, etc.)

Perhaps a fixed syntax should be defined for return values so as to
simplify parsing of output from a command.

A command would either succeed and produce an output or
fail and produce an error. In tcl this would mean using the
primary return value + exceptions. Ditto for Python I guess.

Actually the output from a command should be divided into
human readable progress output and the output from
the command as to be parsed by a program.

Here the human readable output is subject to change without
notice and the computer parsable output is part of the command
definition.

>  > Really what this is is a remote procedure call interface implemented
>  > in some ad-hoc minimalistic way.
>
>  Yup. Where to start? List of possible output from OpenOCD? How big a
>  change would be needed? Or should a completely new interface be written
>  onto the API instead of trying to add another output alternative to
>  the current interface?

I'd start by writing up the protocol.

On Tue, Apr 22, 2008 at 5:26 PM, Ville Voipio <Ville.Voipio at kpatents.com> wrote:
>
>
>
> > That should be straightforward. I can see why you'd want to toss
>  > out telnet, less to worry about + also, you'd want to define the
>  > input/output syntax a bit more so you can, programatically, tell
>  > whether or not a command failed.
>
>  Exactly. The input syntax is not a problem, as it is well-defined
>  and quite well-documented. It is easy to form all sorts of commands.
>  The output syntax might benefit from a bit more uniform format,
>  but even that is not a big problem.
>
>  Of course, as we have error codes for everything, printing those
>  would not hurt... (But then the absolute numbers for the codes
>  have to be fixed to avoid version catastrophes.)

The error codes are not really used and propagated in a robust
fashion.

I'm thinking it would be better to show a success/failure +
the error texts in a  parseable form.


>  Anyway, I'd be perfectly happy with the telnet interface coming
>  to the console without any changes.

I'm thinking that is a bad idea. Look to GDB. It defines a console
meant for humans and the MI interface meant for computers.

If we do anything, I'd like to see a parallel to the MI interface where
the different types of output are clearly differentiated in a way
that's easy to parse for a python/tcl script.

Really what this is is a remote procedure call interface implemented
in some ad-hoc minimalistic way.

I'd like to see this done in a bit more robust fashion that
passing the telnet interface to a python/tcl script. Removing the TCP/IP
communication is one part of making it  more robust, but more
significantly the protocol should be defined more clearly(like GDB MI
only simpler :-)

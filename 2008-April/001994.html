Øyvind Harboe wrote:
>>  Another thought from having walked through the code in ft2232.c is that a lot of
>>  the logic is in common between the devices and could be moved out into
>>  configuration files rather than coded up as a set of code where the main
>>  difference is a set of PIN mappings. If I have to add another variation of pin
>>  mappings I may just tackle that myself if there isn't any other planned stuff in
>>  that area.
> 
> IMHO, a file in interface/xxx.cfg and C code are pretty much indisinguishable
> from the users point of view.

The difference is that the user has to get and recompile openocd to add support 
for a varient on a device - if it is in configuration files then just add the 
files into an existing built. There are a lot of openocd binary builds out there 
and it looks like board vendors are including their own varients - having this 
stuff data driven makes that situation more managable IMHO.

Having just waded through a pile of forum postings the majority of which are 
building and configuring openocd to match how the vendor pre-built an older 
version on linux, win32, etc setups it seemed like something worth considering; 
however I'm in no rush to do that given that I have no problems with compiling 
openocd myself so for me at least I do also agree .c and .cfg make little 
difference.

Ville Voipio wrote:
> (especially section 3.4). The flash programming algorithm is very  
> straightforward and lends itself decently to SVF. The only worrying  
> thing is that the JTAG frequency must be 350+-100 kHz, which may be a  
> difficult specification to meet with some programming HW.

Hmm, that's messy :-( Seems that we'd be better off by running the Flash
code with a local clock. Fortunately, it seems that we should be able to
just load the code into RAM (with JTAG) and then let the CPU execute it.

What's a bit more worrying is this:

http://www-k.ext.ti.com/srvs/cgi-bin/webcgi.exe?Company={5761bcd8-11f5-4e08-84e0-8167176a4ed9},kb=dsp,case=37917,new

| 2. You ca not put a MSP430 into a JTAG chain together with other devices.

However, section B.2 of SLAA149D claims:

| The MSP430 device must be the first device in the JTAG chain (because of
| clocking via TDI and JTAG fuse check sequence).

Which probably wasn't meant as "it's the first ... and only".
Unfortunately, the SetTCLK/ClrTCLK hack is indeed all over the place,
and not just in the Flash programming sequences.

Helpfully, the KnowledgeBase entry's disclaimer states:

| In the event that information received from the TI Semiconductor
| Technical Support KnowledgeBase conflicts with written documentation
| that was supplied by TI or originated from TI, then the information
| in the written documentation prevails and should be used.

This needs further clarification.

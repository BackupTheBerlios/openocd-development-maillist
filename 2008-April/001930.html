Hello Edwin,

Cituji Edwin Olson <eolson at mit.edu>:
> We should avoid using any commands that could be interpreted as
> synonyms of other commands. E.g., we should not have commands named
> both "write" and "program". We *could* attach meanings to these, but
> it's frustrating to new users to have to learn our arbitrarily-attached
> meanings to words they already "know".

I agree.

> "erase_and_write myimage.out"

Seems ok to me.

> (While we're at it, I might also suggest making the write command a bit
> more general and getting rid of write_bank. I'm imagining an all-in-one
> "write" command of the form:
>
>    flash write <file> [fileoffset] [destination-address]

Not a good idea, this was discussed before, there may be several flash  
banks handled by several drivers and the current state allows for a  
flash not mapped to common address space. Hence it is good there are  
means to operate on flash memory by specifying particular bank.

> I'd also get rid of "erase_address", since devices don't support
> arbitrary erasure at an address granularity. (If this command currently
> does a RMW cycle, then the command should be more transparently
> named... )

This is for users convenience, some MCUs contain flash compound of  
erase sectors of different size (e.g. LPC2xxx) and it is really  
inconvenient to erase memory based on sector numbers and write its  
content to a place specified by address just after that - really  
confusing.

I think it is good to have convenient means of doing the usual stuff,  
as long as there is possibility of access on lower level (e.g. specify  
flash bank number by hand)

   Pavel

On Monday 07 April 2008 23:12:35 Michael Schwingen wrote:
> Dominic Rath wrote:
> > Ideally, it should only be asserted once, when the OpenOCD first
> > connects. It could further be asserted in case the OpenOCD got out of
> > sync, or when there's a chance of loosing sync as is the case with
> > ARM7TDMI-S and reset-while-halted.
> > Are power outages during debug really that much of a problem?
>
> Yes. I tend to power-cycle the target quite often during bootloader
> development, since that is the only way to make sure you have
> initialized everything. The BDI2000 at work catches this just fine (it
> does have a target power-detect input, so it can react directly and
> re-init when power is up).
>
> For example, if I mess up SDRAM mode-register setup, the board will run
> fine through any number of resets - until I power-cycle, so it is
> definitely nice if I do not have to restart OpenOCD.

Alright, but there you have dedicated hardware that catches the power-off (or 
rather power-on, I guess). The only chance we have would be to periodically 
poll the target power state (some FT2232 interface would support this), but 
there's currently no mechanism for something like that.

Maybe a failed IR scan (multiple failed IR scans?) can be used to identify the 
need to reset via TRST?

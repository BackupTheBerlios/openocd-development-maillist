Spencer Oliver> We have just changed the stm32 flash driver buffer to
16k (was 8k).
Spencer Oliver> Is this worth doing on other flash drivers ??

Actually, I think coming up with a "common architecture" to do ARM flash
would be better, or is more important.

By that I specifically mean:

1)  Download and algorithm to the target.

2a)  That algorithm talks via the DCC to the host.
     - What ARM does not have a DCC today?
     - I believe it is safe to assume it is always present.
    - if not - we define some other means.

2b)  A similar *SAME* technique can be used for other targets.
    The idea is we create *one* technique.

3)   That algorithm should/could be *target* specific and can *REPORT*
       how much buffer memory is present.

4)   That algorithm should/could use a *COMMON* set of commands
    That is *common* to *ALL* targets.

5)   how exactly each target responds to "erase" - is a target problem.
    Nothing more, and is target specific -
    however the command and protocol is identical.

    How that algorithm responds to "program" -   is target specific
    however the command and protocol is identical.

6)  How the host side talks to the "target flash programer" can be
identical.
    ie: We can create a *target* specific flash function struct?
    Much like we have the "target" structure with function pointers?

-Duane.
